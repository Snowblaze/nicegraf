\doxysubsection{Detailed Description}
This section contains the documentation for the core nicegraf routines, structures and enumerations.\hypertarget{group__ngf_core-remarks}{}\doxysubsubsection{General Remarks}\label{group__ngf_core-remarks}

\begin{DoxyItemize}
\item The library is currently not intended to be linked dynamically.
\item When nicegraf\textquotesingle{}s C headers are included from C++, all global functions within them are automatically declared to have C linkage. Additionally, they are declared to be noexcept.
\end{DoxyItemize}\hypertarget{group__ngf_object-model}{}\doxysubsubsection{Object Model}\label{group__ngf_object-model}
nicegraf objects, such as images, buffers, render targets, etc., are represented using opaque handles. The objects are created and destroyed explicitly by the application, and it is the responsibility of the application to ensure a correct order of destruction.\hypertarget{group__ngf_error-reporting}{}\doxysubsubsection{Error Reporting}\label{group__ngf_error-reporting}
Most nicegraf routines report their completion status by returning an \mbox{\hyperlink{group__ngf_ga0cf1ffd267fbb5a398934dc6827053ad}{ngf\+\_\+error}}, and write their results to out-\/parameters. The returned value is a generic error code. Detailed, human-\/readable information about errors may vary from platform to platform; nicegraf reports it by invoking a user-\/provided callback function (see \mbox{\hyperlink{structngf__diagnostic__info}{ngf\+\_\+diagnostic\+\_\+info}}). The callback function must accept the diagnostic message type (see \mbox{\hyperlink{group__ngf_ga73f485c050416fe7ae3f5286cc418295}{ngf\+\_\+diagnostic\+\_\+message\+\_\+type}}), an arbitrary void pointer (the value of which the user may specify when providing the callback), a printf-\/style format string, and an arbitrary number of arguments specifying the data for the format-\/string.\hypertarget{group__ngf_host-memory-management}{}\doxysubsubsection{Host Memory Management}\label{group__ngf_host-memory-management}
By default, nicegraf uses the standard malloc/free to manage host memory for internal purposes. The client may override this behavior by supplying custom memory allocation callbacks (see \mbox{\hyperlink{structngf__allocation__callbacks}{ngf\+\_\+allocation\+\_\+callbacks}}).\hypertarget{group__ngf_gpu-memory-management}{}\doxysubsubsection{GPU Memory Management}\label{group__ngf_gpu-memory-management}
nicegraf internally manages GPU memory for all backends. It is currently not possible for clients to override this behavior and do their own GPU memory management. \doxysubsection*{Data Structures}
